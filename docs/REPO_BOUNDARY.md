# Repo Boundary

Purpose: define what belongs in the Slomix git repo, what stays local, and how to reason about borderline files before committing them.

This is the foundation document. It is intentionally simple.

## The One Rule

Track a file if a clean checkout needs it to understand, build, run, or deploy Slomix.

Do not track a file if it is:
- a secret
- generated by runtime
- machine-local state
- a backup or scratch artifact
- a large external input that can be recreated or fetched elsewhere

## Borderline Areas That Need Explicit Policy

These should never be committed "by accident". If we keep them in git, it must be intentional.

- `website/static/modern/`
  Current question: are built frontend artifacts part of deployment, or should deploy build them from source?
- `docs/reports/`
  Some are useful decision records, some are temporary research dumps.
- root-level one-off files
  Examples: screenshots, throwaway Lua copies, debug scripts, temporary markdown plans.

## Top-Level Mental Map

If you are lost, start here:
- `bot/`: Discord bot runtime, polling, parsing, publishing
- `analytics/`: synergy analytics dependency loaded by the bot
- `website/backend/`: FastAPI API
- `website/frontend/src/`: React source
- `website/js/`: current browser runtime used by the existing site
- `greatshot/`: demo and highlight pipeline
- `proximity/`: proximity capture and import pipeline
- `vps_scripts/`: game-server-side webhook and trigger scripts
- `scripts/`: deploy, smoke, repair, and ops helpers
- `migrations/`: schema history
- `docs/`: architecture, incidents, runbooks, decisions

## Commit Filter

Before committing, ask only these four questions:

1. Would a new machine need this file to run or understand Slomix?
2. Does this file contain a secret or machine-specific value?
3. Is this file generated by runtime, tests, or a build?
4. If this file disappeared, could we recreate it cheaply?

If the answers are:
- yes, no, no, no: track it
- no to question 1: probably do not track it
- yes to question 2 or 3: do not track it
- yes to question 4: default to not tracking it unless it is a migration or important decision record

## Immediate Practical Policy

- Develop on `samba`; do not treat `slomix_vm` as the source of truth.
- Deploy exact commits to the VM instead of hand-copying mixed files.
- Keep templates in git, keep live secrets out of git.
- Prefer code, migrations, and deploy scripts over runtime dumps and screenshots.
- When unsure, classify the file in this document before committing it.
