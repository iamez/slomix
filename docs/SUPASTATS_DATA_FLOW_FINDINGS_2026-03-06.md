# Supastats Data Flow Findings 2026-03-06

## Scope

This note traces where session stats are collected, transformed, and rendered inside the Slomix repo, with a specific goal:

- determine whether the `supastats*.png` screenshots are generated by this repo
- identify where canonical fields come from
- identify which screenshot fields are upstream-only

## Conclusion

The repo contains the canonical Slomix ingestion + database + website + Discord rendering pipeline, but there is no in-repo generator for the `supastats9.png` through `supastats13.png` screenshot set.

Those screenshots appear to come from an upstream external stats system fed by the ET:Legacy Lua stats exporter, not from the Slomix website or Discord embed renderer.

## In-Repo Canonical Pipeline

### 1. Stats are collected from ET:Legacy Lua output

- `docs/reference/oksii-game-stats-web.lua` documents the upstream stats exporter configuration and remote submission flow.
- It explicitly defines `api_url_submit`, `api_url_matchid`, and `json_filepath` as configuration for external submission and optional local JSON dump.
- The script builds a JSON payload and submits it with `curl -X POST` to `configuration.api_url_submit`.

Evidence:

- `docs/reference/oksii-game-stats-web.lua:202-220`
- `docs/reference/oksii-game-stats-web.lua:2721-2910`

This is the strongest repo-local evidence that raw session stats are first sent to an external endpoint.

### 2. Slomix also captures round-end timing through its own Lua webhook

- `vps_scripts/stats_discord_webhook.lua` is a separate ET:Legacy Lua script.
- It exists to send round metadata and accurate timing to Discord/Slomix because the primary stats file has timing issues on surrender.
- The header explicitly says the c0rnp0rn/oksii stats script provides per-player stats, while the Slomix webhook provides timing/team capture.

Evidence:

- `vps_scripts/stats_discord_webhook.lua:1-61`

### 3. Slomix ingests parsed player stats into PostgreSQL

- `bot/community_stats_parser.py` parses c0rnp0rn-format stats files.
- It treats `kill_assists` as a real parsed field from the upstream stats format.
- Parsed values are inserted into `player_comprehensive_stats`.

Evidence:

- `bot/community_stats_parser.py:35-67`
- `postgresql_database_manager.py:2587-2638`

### 4. Session-level canonical stats are aggregated from PostgreSQL

- `bot/services/session_data_service.py` resolves which rounds belong to a session.
- `bot/services/session_stats_aggregator.py` aggregates canonical session values from `player_comprehensive_stats` and weapon tables.
- This includes `kills`, `deaths`, `damage_given`, `damage_received`, `kill_assists`, and weighted `dpm`.

Evidence:

- `bot/services/session_data_service.py:64-181`
- `bot/services/session_stats_aggregator.py:53-135`

### 5. Website session payload is derived from canonical DB values

- `website/backend/routers/api.py` builds the session-detail player payload.
- It computes:
  - `alive_pct` from dead/playtime
  - `played_pct`
  - `played_pct_lua`
  - `supastats_tmp_pct`
  - `supastats_tmp_ratio`
- It keeps legacy `tmp_pct` as alive%.

Evidence:

- `website/backend/routers/api.py:3120-3361`

### 6. Website renders session detail from API JSON, not pre-rendered screenshots

- `website/js/session-detail.js` only normalizes API fields and renders HTML/Chart.js UI.
- It does not save PNGs, generate screenshots, or export the supastats image set.

Evidence:

- `website/js/session-detail.js:168-226`
- `website/js/session-detail.js:2224-2542`

## Why The Screenshot Mismatches Happened

The repo itself contains evidence that the screenshot fields were being mapped onto canonical fields only as a best-effort comparison, not because a true field contract existed.

Artifacts:

- `docs/archive/root-artifacts/supastats/supastats_diff_2026-03-03.csv`
- `docs/archive/root-artifacts/supastats/supastats_diff_2026-03-03_summary.txt`
- `docs/archive/root-artifacts/supastats/supastats_diff_2026-03-03_superboyy.csv`

These files explicitly record:

- `A` was mapped to canonical `kill_assists`
- `TMP` was mapped to `1 - dead_time/play_time`
- `EFFORT`, `EXPECTED`, and `PERF` had no canonical DB column

Evidence:

- `docs/archive/root-artifacts/supastats/supastats_diff_2026-03-03_summary.txt:13-28`
- `docs/archive/root-artifacts/supastats/supastats_diff_2026-03-03.csv:5-13`
- `docs/archive/root-artifacts/supastats/supastats_diff_2026-03-03_superboyy.csv:5-13`

That means the original mismatches were mostly contract/mapping errors, not proof that the DB ingestion was wrong.

## Field Status

### Canonical and repo-backed

- `DG`
- `DR`
- `K`
- `G`
- `SK`
- `D`
- `KD`
- per-map `DPM`
- per-map `Kills`
- team totals

### Canonical but renamed/fixed for screenshot parity

- `TMP`
  - screenshot behavior aligns with played-time semantics
  - repo now exposes `supastats_tmp_pct` / `supastats_tmp_ratio`
  - legacy `tmp_pct` remains alive%

Primary evidence:

- `website/backend/routers/api.py:3300-3361`

### Unresolved upstream-only

- `A`
  - not equal to canonical `kill_assists` on validated sessions
- `EFFORT`
- `EXPECTED`
- `PERF`

These should not be treated as Slomix calculation failures until the upstream generator/formulas are traced.

## Practical Meaning

If the goal is to validate Slomix session correctness:

- trust the DB-backed fields and session validator
- treat screenshot `TMP` as the new `supastats_tmp_*` contract
- do not treat screenshot `A` as `kill_assists`
- do not validate `EFFORT / EXPECTED / PERF` against Slomix until their formula source is imported or documented

## Next Required Trace

The missing source is outside the current repo boundary:

- the external service behind `api_url_submit`
- or the external website/process that turns submitted match data into the supastats-style visual tables

Without that upstream source, `A`, `EFFORT`, `EXPECTED`, and `PERF` remain external metrics.
